# SANDWICH : Sequencing ANalysis and Data Workflow for Integrated Characterisation of microbiomes

## Overview

This pipeline provides an end-to-end workflow for shotgun metagenomic analysis of paired-end sequencing data from food metagenomes. It is designed to run on the QIB HPC cluster using SLURM array jobs, with a modular step-by-step structure that allows users to control which analyses are run and to verify outputs between steps before proceeding.

The pipeline covers host read removal (human and food matrix), quality control, taxonomic profiling, strain-level analysis, and functional annotation, with aggregated reports generated at each stage.

---

## Pipeline Structure

```
01.run_qc.sh              → Host removal + quality trimming
01.check_qc.sh            → Verify QC outputs before proceeding

02.run_taxonomic.sh       → Taxonomic profiling (MetaPhlAn4, Kraken2, SameStr)
02.check_taxonomic.sh     → Verify taxonomic outputs before proceeding

03.run_functional.sh      → Functional annotation (HUMAnN3, ARG, virulence factors)
03.check_functional.sh    → Verify functional outputs before proceeding
```

Each step is submitted independently. Check scripts must be run manually between steps to verify outputs and catch failures early.

## How to Use the Pipeline on the QIB Cluster

### 1. Clone or copy the pipeline to your working directory

```bash
cp -r /path/to/pipeline /qib/scratch/users/<username>/my_project/
cd /qib/scratch/users/<username>/my_project/
```

### 2. Generate a sample sheet

Run the sample sheet generator pointing to your directory of paired-end reads:

```bash
bash generate_samplesheet.sh /path/to/reads _R1_001.fastq.gz _R2_001.fastq.gz
```

This produces a `samples.tsv` file with three columns: `sample_id`, `read1`, `read2`. Check it before proceeding.

**Multi-matrix projects:** if your project contains samples from different food matrices (e.g. some from dairy, some from fish), you can add an optional fourth column `food_host` to the sample sheet to specify the host reference per sample. The `generate_samplesheet.sh` script does not add this column — open the file in Excel, add the column manually, and save as tab-separated before running the pipeline.

```
sample_id    read1                read2                food_host
sample1      /path/sample1_R1     /path/sample1_R2     cow
sample2      /path/sample2_R1     /path/sample2_R2     salmon
blank1       /path/blank1_R1      /path/blank1_R2      none
sample3      /path/sample3_R1     /path/sample3_R2
```

Valid values for `food_host` are: `cow`, `salmon`, `soybean`, `pig`, `spinach`, `none`. If the column is absent or the value is empty for a sample (as for `sample3` above), the pipeline falls back to `FOOD_HOST_REF` from `config.sh`. Use `none` explicitly for blank samples to skip food host removal.

### 3. Edit the config file

Open `config.sh` and set the paths and options for your project:

```bash
nano config.sh
```

Key settings to review:

- `SAMPLE_SHEET` — path to your samples.tsv
- `OUT_DIR` — where results will be written
- `LOG_DIR` — where SLURM logs will be written
- `TYPE` — sequencing type: `illumina` or `mgi`
- `REMOVE_HUMAN` — set to `true` to remove human reads first (recommended for all food microbiome studies to control for operator contamination); set to `false` only if you are certain no human contamination is possible
- `FOOD_HOST_REF` — food or environmental host to remove after human decontamination; options: `cow`, `salmon`, `soybean`, `pig`, `spinach`; set to `none` to skip this step
- Module switches (`RUN_METAPHLAN`, `RUN_KRAKEN`, `RUN_HUMANN`, `RUN_ARG`, `RUN_VIRULENCE`, `RUN_SAMESTR`)

Database paths are pre-configured for the QIB cluster and should not need to be changed unless databases are updated.

### 4. Run the pipeline step by step

**Step 1 — QC and host removal:**
```bash
bash 01.run_qc.sh
```
Monitor jobs with `squeue -u <username>`. Once complete:
```bash
bash 01.check_qc.sh
```
Review the summary. If all samples pass, proceed to step 2.

**Step 2 — Taxonomic profiling:**
```bash
bash 02.run_taxonomic.sh
```
Note: if `RUN_SAMESTR=true`, a second job (`02b`) will be submitted automatically with a dependency on the array job. This job can take up to 72 hours. If `RUN_METAPHLAN=true`, a merge job (`02c`) will also be submitted automatically. Once all jobs are complete:
```bash
bash 02.check_taxonomic.sh
```

**Step 3 — Functional annotation:**
```bash
bash 03.run_functional.sh
```
A merge job (`03b`) will be submitted automatically with a dependency on the array job. Once complete:
```bash
bash 03.check_functional.sh
```

## Running the Rmarkdown Reports

Each pipeline stage produces an aggregated output folder (`04_aggregated/`) that contains everything needed to render the corresponding report. The Rmarkdown templates are automatically copied into the relevant subfolder by the merge jobs, so users only need to open that folder and follow the steps below.

### 1. Prepare the metadata file

Before rendering any report, you must provide a metadata file at:

```
results/04_aggregated/metadata.tsv
```

This file must be tab-separated with the following three columns:

| Column | Description |
|--------|-------------|
| `sequence_id` | Sample ID matching the filename without extensions (e.g. `sample1`) |
| `sample_name` | Display name used in plots (e.g. `Patient 1 - Week 0`) |
| `sample_group` | Group label for colour coding; use `blank` for negative controls |

Example:

```
sequence_id     sample_name     sample_group
sample1_S1      Patient1_W0     treatment
sample2_S2      Patient2_W0     treatment
blank1_S10      Blank1          blank
```

The `sample_group` column drives colour coding throughout all reports. Any number of biological groups can be used. Blank samples must be labelled exactly as `blank` (lowercase) as they are handled separately for decontamination.

### 2. Render the QC report

Once `01.check_qc.sh` passes and `metadata.tsv` is in place:

```bash
cd results/04_aggregated/
Rscript -e "rmarkdown::render('01_qc/01_qc_report.Rmd')"
```

This produces `01_qc_report.html` in the `01_qc/` subfolder.

### 3. Render the taxonomic profiling report

Once `02.check_taxonomic.sh` passes:

```bash
cd results/04_aggregated/
Rscript -e "rmarkdown::render('02_taxonomic_profiling/02_taxonomic_report.Rmd')"
```

This produces `02_taxonomic_report.html` in the `02_taxonomic_profiling/` subfolder and also saves a decontaminated phyloseq object at `02_taxonomic_profiling/ps_clean.rds` for downstream use.

### 4. Adjusting report parameters

Each Rmarkdown file contains a `USER-ADJUSTABLE PARAMETERS` section near the top of the setup chunk. Key parameters that can be modified before rendering:

**QC report (`01_qc_report.Rmd`):**
- `MIN_READS` — minimum read count threshold for flagging low-depth samples (default: 10,000)

**Taxonomic report (`02_taxonomic_report.Rmd`):**
- `RATIO_THRESHOLD` — decontamination ratio filter threshold (default: 0.1)
- `SPAN_THRESHOLD` — decontamination span filter threshold (default: 0.5)
- `UNCLASSIFIED_THRESHOLD` — flag samples with more than this proportion unclassified (default: 0.75)
- `TOP_N_TAXA` — number of top taxa shown in composition barplots (default: 15)

## Notes and Known Limitations

- Host removal runs in two sequential steps: human removal first (controlled by `REMOVE_HUMAN`), then food host removal (controlled by `FOOD_HOST_REF`). Either step can be disabled independently. Both steps always produce a JSON log — skipped steps write a placeholder with zero counts so the summary CSV is consistent across all configurations.
- For projects with multiple food matrices, the food host reference can be overridden per sample by adding a `food_host` column to the sample sheet. The per-sample value takes priority over `FOOD_HOST_REF` in `config.sh`. Samples with an empty value in the column fall back to the config default.
- At least one of `REMOVE_HUMAN=true` or `FOOD_HOST_REF` being set is strongly recommended. Running neither step is permitted but will produce a warning.
- SameStr requires MetaPhlAn4 to be enabled (`RUN_SAMESTR=true` requires `RUN_METAPHLAN=true`). The pipeline will check this and exit with an error if misconfigured.
- HUMAnN3 runs its own internal MetaPhlAn step and is independent of the `RUN_METAPHLAN` setting.
- Diamond outputs (CARD, VFDB) may be empty for some samples if no hits are found. This is expected and is reported as `NO HITS` rather than a failure in the check script.
- All functional annotation steps use R1 reads only for computational efficiency.
- The Python merging script for Diamond results runs inside a Singularity container (`containers/merging_results.sif`). This container must be built once before running step 3 (see below).
- The decontamination step in the taxonomic report requires at least one sample labelled `blank` in the metadata. If no blanks are present, the step is skipped and all samples are retained.

### Building the Singularity container

```bash
singularity build containers/merging_results.sif containers/merging_results.def
```

This only needs to be done once per installation.

## Troubleshooting

- Check SLURM logs in `$LOG_DIR` — logs are named by job name, array ID, and task ID for easy identification.
- If a check script reports missing files, look at the corresponding `.err` log for that sample to identify the error.
- If the SameStr population job (`02b`) fails, it can be resubmitted independently once the issue is resolved:

```bash
sbatch --partition=qib-long --time=72:00:00 --cpus-per-task=30 \
    --mem-per-cpu=5000M modules/02b_samestr_population.sh
```

- If the MetaPhlAn merge job (`02c`) fails, it can be resubmitted independently:

```bash
sbatch --partition=nbi-short --time=01:00:00 --cpus-per-task=1 \
    --mem-per-cpu=8G modules/02c_metaphlan_merge.sh
```
